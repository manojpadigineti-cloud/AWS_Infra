[6] Working dir/staging/repo difference?
Working directory contains modified tracked/untracked files visible in git status red. Staging area (index) holds git add changes ready commit shown git diff --staged green. Repository .git/objects stores committed history immutable blobs/trees/commits accessible git log.

[7] HEAD in Git?
HEAD is movable pointer to current commit, either branch label (attached HEAD) or direct SHA (detached). Visualize git log --oneline --graph --decorate --all shows HEAD@{?} reflog positions. Detached during checkout SHA or rebase—create branch before new commits to avoid loss.

[8] Tracked vs untracked files?
Tracked files known to Git via prior add/commit, changes show git status modified. Untracked new files not added, listed red unless .gitignore. List git ls-files --others --exclude-standard, clean with git clean -fdx.

[9] git init internally does?
Creates .git/ directory with HEAD (ref: refs/heads/master), config, description, hooks/, info/, objects/ (empty), refs/heads/ refs/tags/. Initializes empty repository ready first git add/commit. Sets user.name/email if configured.

[10] Default branch and change?
Default changed to main from master 2020 inclusivity. Change locally git branch -M main, remote git push -u origin main, GitHub Settings→Branches→Change default. Apply protection rules immediately.

[11] Purpose of .git directory?
Core repository database: objects/ (SHA1 blobs/trees/commits/packfiles), refs/ (branches/tags/ remotes/), HEAD, config (user/repo settings), index (staging), logs/ (ref history), hooks/ (pre-commit etc), info/exclude.

[12] Check current branch?
git branch shows list with * current, git status "On branch main", git rev-parse --abbrev-ref HEAD scriptable. git symbolic-ref HEAD full ref.

[13] What happens git add .?
Stages all changes recursively from current directory: modified tracked files, new untracked (gitignore respected). Creates index entries blobs .git/index. Use git add -A entire repo, -p interactive hunks.

[14] Bare repository?
Repository without working directory, only .git/ contents for server-side push/pull. git clone --bare URL, used central hubs like GitHub mirrors. No checkout, git push --mirror sync.

[15] git pull vs git clone?
git clone URL creates full repo copy with history + working directory first time. git pull = git fetch origin + git merge (or rebase) updates existing clone. Avoid pull shared feature branches.

[16] Fast-forward vs three-way merge?
Fast-forward linear when target ahead source: git merge --ff-only feature integrates without merge commit. Three-way needed diverged histories: common ancestor + source + target → new merge commit parents. Always --no-ff enterprise traceability.

[17] git cherry-pick?
Apply commit(s) from one branch to current: git cherry-pick abc123def copies patch metadata. Conflicts resolve add --continue, -x adds source commit message. Use for hotfixes main without full merge.

[18] git revert vs git reset?
git revert abc123 creates new commit undoing changes safe for shared/public branches. git reset --hard abc123 rewrites history moves HEAD dangerous shared repos. Revert PRs, reset local feature branches.

[19] Types git reset?
--soft HEAD~2 keeps changes staged HEAD moves. --mixed (default) unstaged working tree clean. --hard discards ALL changes to target commit. ORIG_HEAD safety git reset --hard ORIG_HEAD.

[20] origin vs upstream vs local?
origin: your remote fork default git remote add origin. upstream: canonical source git remote add upstream ORIGINAL.git. Local branches no remote prefix. git remote -v rename.

[21] Detached HEAD state?
git checkout abc123 points HEAD direct commit SHA not branch label. New commits orphaned when switch away—recover git reflog create branch. Happens rebase/checkout tag/amend.

[22] Rename branch locally/remote?
Local: git branch -m old-branch new-branch. Remote: git push origin -d old-branch; git push -u origin new-branch. Default branch: GitHub Settings change.

[23] Squash commits before merge?
git rebase -i HEAD~5 → change pick to s squash combine messages. git merge --squash feature → single commit. GitHub UI squash merge option.

[24] Undo last commit keep changes?
git reset --soft HEAD~1 reverts commit keeps staged. --mixed HEAD~1 unstaged working tree intact. git commit --amend edit message.

[25] git diff variations?
git diff: unstaged working vs staging index. git diff --staged: staging vs HEAD commit. git diff HEAD: working vs last commit. --name-only files only.

[26] git reflog usage?
Records all HEAD position changes 90 days: git reflog view git reset --hard HEAD@{5}. Recover reset/delete/rebases. git log -g reflog order.

[27] Recover deleted branch?
git reflog | grep branch-name → SHA git checkout -b recovered SHA. git branch -D local-only deleted remote persists origin/branch.

[28] git bisect binary search?
git bisect start → git bisect bad (current) → git bisect good v1.0 → git bisect run npm test automates. Manual git bisect good/bad halves history.

[29] git blame practical?
git blame -L 42,42 src/main.go line author/date. IDE GitLens. -C copy detection across files.

[30] Resolve rebase conflicts?
git rebase main feature → conflict markers edit files → git add → git rebase --continue. Skip git rebase --skip, abort --abort return original.

[31] Annotated vs lightweight tags?
Lightweight git tag v1.0 SHA pointer. Annotated git tag -a v1.0 -m "Release 1.0" full object signed GPG verifiable info.

[32] Sign commits/tags GPG?

[33] git worktree multiple checkouts?

[35] git subtree vs submodule?

[36] Clean untracked files safely?

[37] git archive export?

[38] How Git stores data internally?
Content-addressable: .git/objects/ SHA1 blobs (files), trees (dirs), commits (pointers metadata), packfiles delta compressed history.

[39] Git compression hashing mechanism?
SHA1 prefix 2 chars/dir objects/12hex blob zlib single → pack delta similar objects 90% smaller. Immutable verify git fsck.
