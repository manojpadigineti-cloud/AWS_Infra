TERRAFORM – ADVANCED INTERVIEW Q&A

Q1. How does Terraform handle state locking, and what happens if the lock is lost mid-apply?
Answer: Terraform uses remote backends (e.g., S3 + DynamoDB) for state locking. If the lock is lost mid-apply, it can cause state corruption or duplicate changes; manual state recovery may be required.

Q2. Explain a real scenario where terraform plan shows no change, but apply still modifies resources.
Answer: Happens due to external changes (drift), dynamic values, or provider bugs where actual state differs from stored state.

Q3. How do you safely manage Terraform state across multiple teams and environments?
Answer: Use remote backend, state locking, separate workspaces or state files, and strict access controls.

Q6. How do you handle secrets in Terraform without exposing them in state files?
Answer: Use external secret managers (Vault, AWS Secrets Manager) and avoid storing secrets directly in state.

Q8. What happens internally when you delete a resource manually from the cloud but not from Terraform?
Answer: Terraform detects missing resource during plan and recreates it unless state is updated.

Q9. How do you design Terraform modules to be reusable without becoming tightly coupled?
Answer: Use input variables, outputs, and avoid hardcoding dependencies.

Q10. Explain depends_on vs implicit dependency — when does Terraform get it wrong?
Answer: Implicit dependency is auto-detected via references; depends_on is manual. Terraform may fail when dependencies are indirect or hidden.

Q12. How do you refactor a Terraform codebase without destroying production resources?
Answer: Use terraform state mv, import, and gradual changes instead of recreating resources.

Q15. Describe a real incident caused by Terraform state corruption. How did you fix it?
Answer: Example: concurrent apply corrupted state; fixed by restoring backup state and reapplying changes.

Q1. What is a Rolling Update in Kubernetes?
Answer:
Rolling Update is the default deployment strategy in Kubernetes where old pods are gradually replaced with new ones.

* Ensures zero downtime
* Allows controlled rollout
* Supports easy rollback by redeploying previous versions

It is best suited for stateless applications.

Q2. What is the Recreate strategy in Kubernetes?
In the Recreate strategy, all existing pods are terminated before new pods are created.

* Simple to implement
* Useful when old and new versions cannot run together
* Causes temporary downtime

It is typically used when applications require exclusive access to shared resources like databases.

Q1. How do you perform maintenance activity on a Kubernetes node?
Maintenance activities like patching or security fixes require making the node unschedulable and draining pods.
Commands:
kubectl cordon <node>
kubectl drain <node> --ignore-daemonsets

This ensures pods are safely moved while avoiding disruption to daemonsets.

1. Q: Why is timestamp standardization critical in Kubernetes logging?
   A: Logs across clusters, apps, and tools use different timezones; without UTC standardization, correlation becomes unreliable. In production, teams enforce UTC to align logs with monitoring (e.g., Prometheus, ELK).

2. Q: What is the impact of stdout vs stderr in container logging?
   A: stdout carries normal logs, stderr carries errors; many pipelines only ingest stdout. In real-world setups (Fluentd/CloudWatch), misconfigured pipelines can completely miss critical failures.

3. Q: Why should you specify container name in `kubectl logs`?
   A: Pods can have multiple containers; default logs may come from the wrong one. In production (sidecars like Istio/Fluentd), always use `-c` to avoid debugging incorrect components.

4. Q: Do Kubernetes pods generate logs directly?
   A: No, containers generate logs; pods are just wrappers. In multi-container architectures (app + sidecar), each container has isolated logs, requiring explicit selection.

5. Q: Why can `kubectl logs` be misleading during failures?
   A: It only shows application logs, not infrastructure issues. Real-world outages often stem from kubelet, networking, or storage—not visible in app logs.

6. Q: What are common non-application causes of pod failures?
   A: Node pressure, CNI/network failures, volume mount errors, kubelet issues. In production, engineers correlate logs with `kubectl describe` and node-level diagnostics.

LOGGING STRATEGY

7. Q: Why is structured logging preferred in production systems?
   A: JSON logs enable parsing, indexing, and querying at scale. Tools like ELK, Splunk, and Loki rely on structured data for efficient filtering and alerting.

8. Q: When should you use plaintext vs structured logs?
   A: Plaintext for small setups (human readability), structured logs for production. Enterprises prioritize consistency to enable automation and observability pipelines.

9. Q: What is the biggest advantage of structured logging?
   A: Machine readability—enables real-time alerting, correlation, and analytics. In production, it powers dashboards and incident response automation.

LOG LEVEL MANAGEMENT

10. Q: Why is proper log level usage important in DevOps?
    A: If everything is INFO, signal is lost in noise. Production systems depend on clear severity separation for alerting and incident prioritization.

11. Q: How should log levels be used in real-world systems?
    A: ERROR → immediate action, WARN → needs investigation, INFO → system visibility, DEBUG → temporary troubleshooting. Mature teams enforce this via logging standards.

12. Q: What happens if log levels are misused?
    A: Alert fatigue, missed incidents, and noisy dashboards. In enterprise environments, poor log discipline directly impacts MTTR and operational efficiency.

KUBERNETES DEPLOYMENT STRATEGIES – INTERVIEW Q&A

Q3. What is Blue-Green Deployment?
Blue-Green Deployment involves running two identical environments:

* Blue → current production version
* Green → new version

Traffic is switched to the Green environment after validation.

Benefits:

* Instant rollback
* Safe production testing
* No user disruption

It is usually implemented using Kubernetes Services or Ingress routing.

Q4. What is Canary Deployment?
Canary Deployment releases a new version to a small subset of users before a full rollout.

Example rollout:
5% → 20% → 50% → 100%

This approach helps monitor:

* Errors
* Latency
* User impact

It is widely used in large-scale microservices architectures.

Q5. What is A/B Testing in deployments?
A/B Testing involves serving different versions of an application to different user groups.

* Group A → Version 1
* Group B → Version 2

It is primarily used for:

* Product experimentation
* Feature validation
* User behavior analysis

Unlike Canary, it focuses more on business outcomes than deployment safety.

KUBERNETES PROBES

1. Q: What is a Readiness Probe?
   A: Checks if pod is ready to receive traffic. In production, failed readiness removes pod from service without restarting it.

2. Q: What is a Liveness Probe?
   A: Checks if container is alive. In production, failure triggers container restart to recover stuck apps.

3. Q: What is a Startup Probe?
   A: Runs only during startup phase. In production, prevents premature liveness failures for slow-start apps.

4. Q: When should you use all three probes?
   A: For production-grade apps with slow startup + runtime health checks. Ensures stability and zero downtime.

5. Q: Why should probes have different endpoints?
   A: Each probe serves a different purpose. In production, using same path can cause false restarts.

KUBERNETES – SCENARIO BASED (PRODUCTION)

1. Q: EKS node disk utilization reached ~85%, SSH is restricted, and even after increasing EBS volume the OS didn’t reflect changes — how do you fix it without node access?
   A: Deploy a privileged pod with hostPath mount, chroot into host, run growpart + filesystem expansion (xfs_growfs/resize2fs). Used in production for zero-access node recovery.

2. Q: Why doesn’t EBS volume auto-expand at OS level after resizing?
   A: AWS expands block device only; partition + filesystem must be manually extended inside OS.

3. Q: Why use a privileged pod instead of SSH?
   A: Secure environments block SSH; privileged pod provides controlled, auditable access via Kubernetes.

4. Q: What risks exist with privileged pods?
   A: Full host access can compromise node; restrict via RBAC and use only for emergency ops.

5. Q: How do you ensure this doesn’t happen again?
   A: Use monitoring alerts (disk usage), automate node bootstrap scripts to auto-expand volumes.

6. Q: What tools are required for disk expansion in Linux?
   A: growpart for partition resizing; xfs_growfs or resize2fs for filesystem expansion.

7. Q: How do you identify correct device and partition?
   A: Use lsblk/df -h to map root filesystem to underlying device (e.g., /dev/xvda1).

8. Q: What is the difference between partition resize and filesystem resize?
   A: Partition resize increases disk allocation; filesystem resize makes OS usable space available.

9. Q: How would you automate this fix in production clusters?
   A: Use user-data scripts, DaemonSets, or AMI baking to auto-expand volumes on boot.

10. Q: How do you validate successful expansion?
    A: Run df -h and ensure new disk size reflects expected capacity.
